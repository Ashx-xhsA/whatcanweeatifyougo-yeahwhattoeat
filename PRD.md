# 产品需求与架构决策文档 (PRD)

**项目名称**: 是啊，吃啥 (原名: Smart Recipe Vault)
**项目愿景**: 一个极度轻量、完全本地化、隐私至上且具有高度个性化沉浸感体验的个人食谱管理与检索引擎。

---

## 1. 核心架构决策

### 1.1 数据管线 (Data Pipeline)
*   **决策**: 放弃使用云端数据库或复杂的后端接口，所有数据均依托本地文件管理，贯彻“本地优先 (Local-first)”原则。
*   **实现**: 编写了 `parse_recipes.py` 脚本，使用 `BeautifulSoup` 直接从标准化的本地 HTML 文件（带 Schema.org 标记）中提取结构化信息（姓名、配料、做法等），并直接将涉及到的图片资源硬拷贝至 `public` 文件夹，最终生成前端直接引用的只读 `src/data/recipes.json`。

### 1.2 技术栈 (Tech Stack)
*   **前端框架**: React + Vite (保证了极速的冷启动和热更新)。
*   **样式方案**: 原生 Vanilla CSS。主动放弃 TailwindCSS，以获得对像素级复古 UI 更好的全局控制权。

---

## 2. 核心功能决策

### 2.1 搜索机制与人工智能 (AI vs. Keyword Search)
*   **决策**: **彻底移除所有内置的 AI 模型与语义搜索。回归纯粹的文本关键字秒切。**
*   **演进过程**:
    1. 最初引入了 `@xenova/transformers` 的本地小模型（22MB）做语义检索。
    2. 发现英文模型无法识别中文（如“主食和肉”）。
    3. 曾尝试替换为多语种本地大模型（>100MB），但由于导致网页极其臃肿、加重浏览器负担，被判定为“本末倒置”。
    4. 坚决不使用外部闭源 API (ChatGPT/DeepSeek) 以保证用户的绝对隐私和 0 运行成本。
    5. **定论**: 最终全部切掉。目前仅使用原生 JavaScript 数组过滤进行急速、多字段文本匹配。

### 2.2 国际化与翻译 (I18n & Translation)
*   **决策**: **零延迟的静态双语数据结构，拒绝运行时 API 翻译。**
*   **实现**: 前端不做任何运行时长句翻译。修改了 Python 打包工具，在前置脚本生成 JSON 时，利用 `deep-translator` 将所有的英文数据全部提前翻译好并追加特定的 `_zh` 字段。
*   **UI表现**: 前端增加了 ENG/中文 Toggle，切换到中文后，菜谱和配料采用“粗体中文 + 半透明原版英文”的一一对应双语呈现。

### 2.3 背景音乐交互 (BGM / Audio)
*   **决策**: 引入轻量级环境音效。
*   **实现**: 依靠原生的 HTML5 `<audio>` 标签与 React State 实现播放控制。采用本地读取 `public/bgm.mp3` 的方式，保证音频文件绝不出网，杜绝流媒体延迟。

---

## 3. 视觉与 UI 设计系统决策

### 3.1 主题风格 (Theme)
*   **决策**: 彻底抛弃现代化的玻璃拟物风格（Glassmorphism），全面倒退（进化）为**《星露谷物语 (Stardew Valley)》**复古像素农场风格。

### 3.2 UI 规则
*   **排版字体**: 全局引用复古游戏像素字体 `VT323`，并强制关闭浏览器的文字平滑（`-webkit-font-smoothing: none`）。
*   **面板与按钮**: 不借助任何图片切图，使用原生的 CSS `box-shadow: inset ...` 仿造出游戏内 UI 的厚重硬木框阴影效果和点击下陷触感。
*   **环境背景**: 将页面底色更换为柔和的春季天空蓝 (`#4fa5e2`)，并编写了专门的 `<SpringPetals />` React 组件，依靠纯 CSS 动画算法自动生成漫天飞扬的粉色樱花瓣，增强农场日子的沉浸感。
*   **细节排版**: 配料列表的红棕色方块（Bullet points）被严格对齐到网格与双语文字第一行的最顶端（使用 `align-items: flex-start` 替换 `display: block`）。
